Problem 3 — Model slug generation bug

  Fix: Used slugify library with uniqueness loop

STEP 9: SUPABASE STORAGE — IMAGE UPLOAD MIGRATION
---------------------------------------------------
Problem:
  - app/api/models/route.ts and app/api/blogs/route.ts were saving images
    to local filesystem (public/uploads/) using fs.writeFile
  - Vercel is serverless — filesystem writes don't persist → images lost on deploy
  - app/actions/model.ts and app/actions/blog.ts had hardcoded
    'http://localhost:3000/api/upload' → breaks on Vercel

Fix:
  - Installed @supabase/supabase-js
  - Created Supabase Storage bucket: "uploads" (public)
  - app/api/upload/route.ts → rewrote to use Supabase Storage SDK
      POST: uploads to uploads/ bucket, returns full public URL
      DELETE: extracts filename from URL, removes from bucket
  - app/api/models/route.ts → removed fs/path imports, uses Supabase Storage
      saves to uploads/models/model-{timestamp}.{ext}
  - app/api/blogs/route.ts → removed fs/path imports, uses Supabase Storage
      saves to uploads/blogs/blog-{timestamp}.{ext}
  - app/actions/model.ts → replaced hardcoded localhost with NEXT_PUBLIC_SITE_URL
  - app/actions/blog.ts → replaced hardcoded localhost with NEXT_PUBLIC_SITE_URL

Result: image full public URL saved to model_image / blog_image columns:
  https://rxczymbnnwjwpohzevpv.supabase.co/storage/v1/object/public/uploads/models/model-xxx.webp

STEP 10: NEXT.JS IMAGE HOSTNAME FIX
-------------------------------------
Problem: next/image threw error — Supabase hostname not whitelisted
Fix: Added remotePatterns to next.config.ts:
  hostname: "*.supabase.co"
  pathname: "/storage/v1/object/public/**"
